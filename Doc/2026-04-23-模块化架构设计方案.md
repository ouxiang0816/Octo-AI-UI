# Octo Client Shell 模块化架构设计方案

> 版本：v1.0 | 日期：2026-04-23

---

## 背景与目标

**背景**：当前项目存在 3 个巨型文件（`ClientShell.tsx`、`OctoBuild.tsx`、`VibeDesign.tsx`），样式、业务逻辑、数据耦合在一起，后续迭代风险高。

**目标**：
- 样式归样式，功能归功能，各自可以独立改动
- Mock 数据可无缝替换为真实 API
- 新增工作流有规范可依，改一个文件即可完成注册
- 文件层级清晰，依赖关系可控

---

## 一、整体架构分层

4 层结构，单向依赖（下层不引用上层）：

```
┌─────────────────────────────────────────────┐
│  App / Layout（应用框架）                    │  ← 组合各模块，管理路由和全局状态
├─────────────────────────────────────────────┤
│  Modules（业务模块）                         │  ← 每个模块独立，内部三层
│  octo-build / vibe-design / assets / ...    │
├─────────────────────────────────────────────┤
│  Workflows（工作流注册中心）                 │  ← 意图识别 + 工作流定义，跨模块共享
├─────────────────────────────────────────────┤
│  Shared（公共框架）                          │  ← 纯展示组件 + 公共 hooks + 全局类型
└─────────────────────────────────────────────┘
```

**每个业务模块内部固定三层：**

```
modules/<module-name>/
├── ui/        ← 只管样式和展示，不含业务逻辑
├── hooks/     ← 只管业务逻辑，不含 JSX
└── api/
    ├── index.ts   ← 接口契约（函数签名，永远不改）
    └── mock.ts    ← Mock 实现，上线换成 real.ts 即可
```

### 关键变化对比

| 问题 | Before | After |
|------|--------|-------|
| 巨型文件 | 3 个（ClientShell/OctoBuild/VibeDesign） | 每个文件职责单一，最大不超过 200 行 |
| Mock 数据位置 | 散落在 `data/` + `features/*/mock-data.ts` | 统一在各模块的 `api/mock.ts` |
| 样式和逻辑混合 | 同一文件里 | `ui/` 和 `hooks/` 强制分离 |
| 反向依赖 | `GlobalSearchModal` import `ClientShell` 内部类型 | 统一从 `shared/types` 取 |
| 新增工作流 | 需要改多个文件 | 只需新建模块 + 在 `registry.ts` 加一行 |

## 三、工作流注册中心 + 意图识别层

### 工作流类型定义

```typescript
// workflows/types.ts
export interface WorkflowDefinition {
  id: string                          // 唯一标识
  name: string                        // 显示名称
  keywords: string[]                  // 规则匹配关键词
  module: React.LazyExoticComponent   // 懒加载入口组件
  outputType: 'canvas' | 'document' | 'report' | 'code'
  description: string                 // 用于 AI 意图识别的描述
}
```

### 工作流注册表

```typescript
// workflows/registry.ts ← 新增工作流只改这一个文件
export const WORKFLOWS: WorkflowDefinition[] = [
  {
    id: 'ui-design',
    name: 'UI 设计',
    keywords: ['界面', '设计', '高保真', 'UI', '页面'],
    module: lazy(() => import('../modules/octo-build')),
    outputType: 'canvas',
    description: '生成高保真 UI 设计稿',
  },
  {
    id: 'user-research',
    name: '用户研究',
    keywords: ['用研', '访谈', '洞察', '人物', '旅程'],
    module: lazy(() => import('../modules/octo-insight')),
    outputType: 'document',
    description: '生成用户研究文档和洞察报告',
  },
  // 后续新增：加一条即可
]
```

### 意图识别流程

```
用户输入
   │
   ▼
① 规则匹配（本地，0延迟）
   │  遍历 WORKFLOWS[].keywords，命中 → 直接返回 workflowId
   │
   │ 未命中
   ▼
② AI 兜底（发后端，异步）
   │  把 WORKFLOWS[].description 列表 + 用户输入发给后端
   │  后端返回 { workflowId, confidence }
   │
   ▼
返回 WorkflowDefinition | null
```

调用方接口固定：

```typescript
const workflow = await detectIntent(userInput)
if (workflow) renderWorkflow(workflow)
```

## 四、API 接口契约设计（Mock 可替换机制）

### 设计原则

`index.ts` 定义函数签名（契约），`mock.ts` 是当前实现，上线时新建 `real.ts` 替换导出，调用方零修改。

### 示例：assets 模块

```typescript
// api/index.ts ← 契约，永远不改
export interface AssetsApi {
  fetchDesignSpecs(): Promise<DesignSpecItem[]>
  fetchIcons(category?: string): Promise<IconItem[]>
  fetchPrototypeComponents(): Promise<PrototypeComponent[]>
}

// api/mock.ts ← 演示/开发用
export const assetsApi: AssetsApi = {
  fetchDesignSpecs: async () => DESIGN_SPECS,
  fetchIcons: async (category) => ICONS.filter(...),
  fetchPrototypeComponents: async () => loadPrototypeComponentLinkMap(),
}

// 开关（切换 Mock → 真实 API 只改这一行）
export { assetsApi } from './mock'   // 演示
// export { assetsApi } from './real' // 生产
```

### 各模块 API 契约一览

| 模块 | 主要契约函数 |
|------|-------------|
| `octo-build/api` | `sendMessage()`, `generateDemo()`, `saveArtifact()` |
| `octo-insight/api` | `generatePersona()`, `generateJourney()`, `fetchResearchDocs()` |
| `vibe-design/api` | `generateCanvas()`, `publishSession()`, `joinCollab()` |
| `assets/api` | `fetchDesignSpecs()`, `fetchIcons()`, `fetchPrototypes()` |
| `skill-market/api` | `fetchSkills()`, `toggleSkill()`, `saveSkillState()` |

### Mock 数据整合归位

| 原文件 | 迁入位置 |
|--------|----------|
| `data/skill-market-data.ts` | `modules/skill-market/api/mock.ts` |
| `data/project-assets-data.ts` | `modules/octo-build/api/mock.ts` |

| `data/shared-artifacts.ts` | `shared/api/mock.ts` |
| `features/assets/mock-data.ts` | `modules/assets/api/mock.ts` |
| `features/vibe/mock-data.ts` | `modules/vibe-design/api/mock.ts` |

## 五、实施优先级（建议顺序）

| 阶段 | 内容 | 风险 |
|------|------|------|
| P0 | 建目录骨架 + 迁移 Mock 数据到各模块 `api/mock.ts` | 低，不改运行逻辑 |
| P1 | 拆分 `ClientShell.tsx`：提取 NavSidebar、skill-market 模块 | 中 |
| P2 | 拆分 `OctoBuild.tsx`：分离 octo-build 和 octo-insight | 中 |
| P3 | 拆分 `VibeDesign.tsx`：panels/modals 拆成独立文件 | 中 |
| P4 | 建立 `workflows/registry.ts` + `intent.ts` | 低，新增文件 |
| P5 | 整理 `shared/` 公共组件层 | 低 |
